# CLAUDE.md

Guidance for AI agents (and humans) working in this repository.

## Project

`xmscore` is a C++ support library for the XMS family of products
(xmsgrid, xmsinterp, xmsmesher, xmsextractor) plus pybind11 bindings
exposed as the `xms.core` Python package. See `README.md` for the source
layout and local doc-build commands.

## Branching policy

Default branch: `master`. The repository uses a trunk-based workflow with
short-lived feature branches and long-lived release branches.

### Working branches

- **Never commit directly to `master`.** Always create a topic branch first.
- Use one of these prefixes, matching the patterns already present in the
  history:
  - `feature/<short-description>` — new functionality
  - `fix/<short-description>` — bug fixes
  - `<username>/<short-description>` — exploratory or personal work
- Keep names concrete and concise (under ~30 characters). Avoid abstract
  nouns. Examples from the history: `feature/ci-artifacts`,
  `fix/xmlog-stdio-for-pybind`, `gagelarsen/update-dev-docs`.
- Branches target `master` via pull request. CI runs on every push and PR
  (see `.github/workflows/XmsCore-CI.yaml`); a green build is expected
  before merge.
- Code review is required before merge. The reviewer should flag style
  drift, duplicated logic, and patterns inconsistent with the rest of the
  codebase.

### Release branches

- Release branches are named `release-<MAJOR>.<MINOR>` (e.g. `release-7.0`,
  `release-6.0`). They are kept alive for backports.
- Backport-only PRs target the relevant `release-X.Y` branch instead of
  `master`. Do **not** rebase or force-push these branches.
- Tags follow semver: `MAJOR.MINOR.PATCH` (e.g. `7.0.7`), with optional
  build metadata (`6.2.4+1`). Tags trigger the release pipeline that
  uploads Conan packages and wheels.

### Branches that are off-limits to manual edits

- `gh-pages` — generated by the documentation workflow. Never push by
  hand; rebuild via `generateDocumentationAndDeploy.sh`.

### Sub-agent work

When delegating work to a sub-agent, the sub-agent creates its own branch
off the current branch. Before merging the sub-agent's branch back, run a
code-review pass to catch duplicated logic and style drift. This applies
to local-only branches too — branch hygiene matters whether or not we
push.

## Documentation policy

Documentation lives in three places and **must be kept in sync with code
changes in the same commit or PR** — no documentation drift.

- **C++ API docs** — Doxygen comments in headers (`xmscore/**/*.h`).
  Conventions: every file has a `\file` block with `\brief` and
  `\ingroup`; every public class has `\class` and `\brief`; every public
  function has at least a `\brief`. Idiom-level recipes belong in
  `Doxygen/CommonIdioms.md`. Build locally with
  `doxygen Doxygen/Doxyfile`.
- **Python API docs** — Sphinx sources in `pydocs/source/`. Adding a new
  public Python symbol means adding (or updating) the matching `.rst`
  page under `pydocs/source/modules/` and wiring it into `index.rst`.
  Build locally with `cd pydocs && make html`.
- **Top-level docs** — `README.md` (source layout, build/test/doc
  commands) and `Doxygen/DoxygenMainpage.md` (module overview, idioms
  pointer). Update these when modules are added, removed, renamed, or
  significantly restructured.

When you change code:

1. If you change a public C++ signature, update the matching Doxygen
   comment in the header.
2. If you change a Python-exposed function or class, update the relevant
   `.rst` page under `pydocs/source/modules/` and any `getting_started`
   examples that reference it.
3. If you add, remove, or rename a module, update `README.md` and
   `Doxygen/DoxygenMainpage.md`.
4. If you change a heavily-used idiom (logging, error handling,
   progress, stream IO), update `Doxygen/CommonIdioms.md`.

A code-only change that requires a doc change is incomplete. Flag missing
doc updates in review.

## Code conventions

- Public C++ lives in the `xms` namespace; macros use the `XM_` prefix.
- Headers use `#pragma once`. Each module folder pairs `Foo.h` /
  `Foo.cpp` / `Foo.t.h`; CxxTest test bodies live at the bottom of
  `Foo.cpp` inside `#if CXX_TEST` blocks.
- Pointer ownership crossing library boundaries uses `BSHP<T>`
  (`boost::shared_ptr`). Companion typedefs (`BSCP`, `BWP`, `BDPC`,
  `BSPC`) are in `xmscore/misc/boost_defines.h`.
- Errors and assertions go through `XM_LOG`, the `XM_ENSURE_TRUE` family,
  and `XM_ASSERT` (see `xmscore/misc/XmError.h` and the
  [Common Idioms](Doxygen/CommonIdioms.md) page).

## Verification before declaring work complete

- C++: build via the CI workflow or locally with the Conan toolchain.
- Python: `python -m unittest discover _package/tests`.
- Docs: `doxygen Doxygen/Doxyfile` and `cd pydocs && make html` should
  both build without new warnings.
